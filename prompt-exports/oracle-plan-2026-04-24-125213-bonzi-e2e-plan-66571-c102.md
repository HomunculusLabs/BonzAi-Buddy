## 1. **Summary**

Implement a targeted, test-only path rather than a refactor: add a deterministic `BONZI_E2E_MODE` branch inside `BonziRuntimeManager`, make Electron able to load built renderer assets without a dev server, add a stable renderer readiness signal plus VRM-load suppression for tests, and add a Playwright Electron harness that launches the real app and drives the existing preload/IPC/UI path. This keeps production behavior unchanged outside env-gated branches and exercises the actual Bonzi stack end-to-end.

## 2. **Current-state analysis**

### Boot path today
- `src/main/index.ts`
  - Creates `assistantService`.
  - Registers IPC handlers.
  - Subscribes to assistant events and forwards them to renderer via `'assistant:event'`.
  - Warms the runtime with `assistantService.getHistory()`.
  - Opens the companion window.

- `src/main/window.ts`
  - Creates the `BrowserWindow`.
  - Loads the renderer from:
    - `ELECTRON_RENDERER_URL` only indirectly via `app.isPackaged === false`
    - built `index.html` only when `app.isPackaged === true`
  - This is the main E2E blocker: Playwright launching an unpackaged built app will still hit the dev URL path.

### Assistant request path today
End-to-end call chain:
1. `src/renderer/app.ts` submit handler calls `window.bonzi.assistant.sendCommand({ command })`.
2. `src/preload/index.ts` forwards with `ipcRenderer.invoke('assistant:send-command', request)`.
3. `src/main/ipc.ts` routes to `assistantService.sendCommand(request)`.
4. `src/main/assistant.ts`
   - validates/truncates input,
   - calls `runtimeManager.sendCommand(normalizedCommand)`,
   - maps proposed runtime actions into `AssistantAction`s,
   - returns `AssistantCommandResponse`.
5. `src/main/eliza/runtime-manager.ts`
   - lazily initializes `AgentRuntime`,
   - configures `localdb` + Bonzi context plugin + model plugin,
   - uses `runtime.messageService.handleMessage(...)`,
   - parses the JSON envelope and emits optional emote events.
6. `src/renderer/app.ts`
   - receives the IPC response,
   - pushes the assistant turn into local renderer state,
   - rerenders the speech bubble.

### Reusable pieces
- `ipc.ts` + `preload/index.ts` already define the canonical bridge; do not add test-only IPC.
- `assistant.ts` already gives the right validation/action-mapping layer; reuse unchanged if possible.
- `runtime-manager.ts` is the correct extension point for a deterministic test mode because it already owns runtime boot, status, history, and reply parsing.
- Existing renderer selectors are mostly sufficient; only a stable “app ready” marker is missing.

### Blocking issues
1. **No test harness/tooling**: no Playwright or Electron E2E script exists.
2. **Launch mode mismatch**: built app cannot be launched reliably in tests because `window.ts` assumes non-packaged means dev server.
3. **Runtime nondeterminism**: `sendCommand()` depends on model output, so CI reliability is poor.
4. **VRM flake risk**: renderer always starts VRM load during hydration.
5. **No isolated user data**: tests would share Electron/localdb state unless overridden.

### Constraints
- Keep the real preload → IPC → main → renderer path intact.
- Do not replace `AssistantService` or IPC with mocks; that would miss the stated proof.
- Electron main process owns runtime state; renderer owns only ephemeral conversation DOM state.
- Persistence is localdb-backed, but the requested E2E scope does not require proving persistence across restarts.
- Renderer already tolerates some out-of-order runtime-status events by caching `pendingRuntimeStatus`; reuse that behavior.

## 3. **Design**

### A. Add a deterministic E2E runtime mode in `config.ts` and `runtime-manager.ts`

**Why this is the right seam**
- It preserves the real `AgentRuntime` boot path.
- It avoids depending on undocumented Eliza JS plugin/model-handler APIs for a custom provider.
- It removes only the flaky part: model inference.

#### `src/main/eliza/config.ts`
Add a new resolved flag:

```ts
interface BonziElizaResolvedConfig {
  // existing fields...
  e2eMode: boolean
}
```

Resolution rules:
- Read `BONZI_E2E_MODE`.
- Treat only `'1'` as enabled for minimal ambiguity.
- Keep existing provider resolution unchanged; the test harness will explicitly set `BONZI_ASSISTANT_PROVIDER=eliza-classic`.

No persistence/schema changes.

#### `src/main/eliza/runtime-manager.ts`
Modify `getOrCreateRuntime()` signature behavior indirectly by including `e2eMode` in `configSignature`, so toggling the flag invalidates cached runtime config.

Add a private helper:

```ts
private buildE2eTurn(command: string): BonziRuntimeTurn
```

Behavior:
- `reply`: deterministic exact string, e.g. `E2E assistant reply for: ${command}`
- `actions`:
  - include one `report-shell-state` action when `command.toLowerCase().includes('shell')`
  - otherwise `[]`
- `warnings`: `[]`
- `emote`: omit

`sendCommand(command)` flow becomes:
1. `await getOrCreateRuntime()` first, unchanged.
2. If `config.e2eMode === true`:
   - skip `runtime.messageService.handleMessage`
   - generate the deterministic envelope via helper
   - run it through existing envelope parsing/sanitization logic so `parseBonziAssistantEnvelope()` remains exercised
   - return the parsed turn
3. Else keep current behavior unchanged.

Important design choice:
- **Do not persist deterministic test turns** in the minimal pass unless the JS `AgentRuntime` typings clearly expose a supported memory-write API.
- Validation task for implementer: inspect `@elizaos/core/node` typings; if there is a first-class memory write method already used elsewhere, persistence can be added later without changing the test contract.

Error handling:
- Runtime init failures still surface exactly as today through `AssistantCommandResponse.ok = false`.
- Deterministic mode does not suppress runtime boot errors.

### B. Make Electron launch the built renderer in tests without a dev server

#### `src/main/window.ts`
Refactor internal renderer-target resolution only; keep `createCompanionWindow(): BrowserWindow` unchanged.

New precedence:
1. If `process.env.ELECTRON_RENDERER_URL` exists, load that URL. This preserves `electron-vite dev`.
2. Otherwise, load `join(__dirname, '../renderer/index.html')`. This is the new build/E2E path and should no longer depend on `app.isPackaged`.

Also add an env-driven renderer query flag:
- If `process.env.BONZI_DISABLE_VRM === '1'`, append `bonziDisableVrm=1`.

Implementation note:
- For `loadURL`, mutate a `URL` object and preserve the existing origin/path.
- For `loadFile`, use Electron’s `query` option; if that proves incompatible with the generated path, validate with `pathToFileURL(...).toString()` + `loadURL` instead.

#### `src/main/index.ts`
Before `app.whenReady()`, add optional user-data override:

```ts
if (process.env.BONZI_USER_DATA_DIR) {
  app.setPath('userData', process.env.BONZI_USER_DATA_DIR)
}
```

Why here:
- `BonziRuntimeManager` derives localdb storage from `app.getPath('userData')`.
- This must happen before service creation/runtime warmup.

No interface changes to `createAssistantService()` are needed.

### C. Add renderer-ready state and VRM suppression

#### `src/renderer/app.ts`
Add a stable readiness marker on the root shell element:

```ts
data-app-ready="loading" | "ready" | "error"
```

State transitions:
- Initial render: `loading`
- After shell state is loaded, history request has settled, and assistant input is enabled: `ready`
- Hard-failure paths only:
  - preload bridge missing
  - `getShellState()` rejected
  - top-level hydration catch
  -> `error`

This should be set in the same code paths that currently enable/disable input and append fatal system messages.

#### VRM suppression
Parse query params once near bootstrap:

```ts
const disableVrm = new URLSearchParams(window.location.search).get('bonziDisableVrm') === '1'
```

If disabled:
- do **not** invoke heavy VRM loading
- provide a local no-op `VrmStageController` with:
  - `dispose(): void`
  - `load(): Promise<void>` that resolves immediately
  - `playBuiltInEmote(): false`
- set `[data-vrm-status]` to a non-error message like `VRM disabled for automated tests`
- keep `vrmErrorEl` hidden and retry hidden

Why keep this in `app.ts` instead of `vrm-stage.ts`:
- Smaller surface area
- No changes to rendering/three.js code paths outside test bootstrap
